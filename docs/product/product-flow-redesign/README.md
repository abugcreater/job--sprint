# Job Sprint 全流程产品重构

日期：2026-07-10

状态：`P0–P5 PASS_WITH_LIMITS`，产品 UI/交互重构与 HTTPS 生产交付已完成；限制为未提交工作树、P8 理论性架构门禁和既有 Vite chunk 告警

## 目标

把 Job Sprint 从模块堆叠的本地工具重构为一条在 Web 与 Android 上都清晰、可达、可恢复的求职闭环：

`准备 → 今日行动 → 对应工作台 → Evidence Gate → 复盘 → 明日第一步`

本轮不是统一换皮。页面结构、移动导航、键盘行为、滚动长度、保存反馈和跨页出口都属于重构范围。

## 已确认的 P0 证据

- Android WebView 为 `384 × 792 CSS px`。
- `/coach` 当前文档高度为 `11171px`，约 `14.1` 个视口。
- 12 次真实上滑只从顶部推进到约 `2899px`，用户需要大量重复手势才能到达底部。
- 页面同时渲染建档、画像、日程、边界、AI 建议、反馈和运行记录，共 9 个二级工作区、100 个交互控件。
- Learning 与 Applications 不在移动底栏，且 More 没有它们的入口，Android 用户无法通过正常界面进入。
- 键盘打开后可用高度从 `792px` 降为约 `487px`，固定底栏继续占用约 `63px`。

这些证据说明问题的根因是信息架构和移动交互，不是简单增加底部 padding 能解决的滚动 bug。

## 互斥方案

### A. 阶段驱动的求职闭环（采用）

- 准备页按“画像、知识边界、今日计划、AI 建议”分阶段。
- 一次只渲染当前阶段，完成后给出明确下一步。
- Today 根据任务类型进入知识、机会、面试或复盘工作台。
- 每个工作台以真实任务、记录和 Evidence 为核心。

### B. 对象中心资料库（暂不采用）

适合大量跨日对象，但当前机会记录仍局限于 sprint 任务，数据模型不足以支持完整资料库。

### C. AI 指令中心（不采用）

操作短，但会隐藏真实状态和人工确认边界；provider fallback 时体验会退化。

招聘 Kanban 不进入候选。当前状态字段不足以构成真实招聘 pipeline，拖拽会制造玩具感。

## 全局信息架构

### 首次使用

1. 导入简历、JD 或面试反馈。
2. 确认目标岗位、经验主线和不可夸大边界。
3. 确认知识边界。
4. 生成今日第一条行动。
5. 处理首条 AI 建议。

### 日常循环

1. Today 只展示当前唯一行动。
2. 根据任务类型进入对应工作台。
3. 在工作台完成真实产出。
4. 回到 Evidence Gate 留下证据。
5. 完成后进入复盘，并明确明日第一步。

### 移动底栏

`今日 / 准备 / 机会 / 面试 / 复盘`

- `/coach` 与 `/learn` 属于“准备”。
- Stats、More、Admin 是低频辅助入口，不占据主循环底栏。
