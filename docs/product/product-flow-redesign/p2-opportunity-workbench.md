# P2：机会工作台

日期：2026-07-10

结论：`PASS_WITH_LIMITS`

## 问题与目标

旧版机会页先展示目标、任务和状态卡，再在页面下方堆叠记录表单。真实机会不是页面主角，移动端也缺少清晰的“列表 → 详情 → 编辑”路径。

本阶段把机会旅程收敛为：

`机会清单 → 事实详情 / 两项对照 → 编辑 → Evidence Gate`

只呈现用户已记录的公司、岗位、状态、JD、沟通反馈和下一步事实；不生成匹配分、成功概率、自动排序或自动投递。

## 方案选择

### 采用：事实型主从工作台

- 桌面端左侧为 340px 机会清单，右侧显示事实详情或两项对照。
- 移动端使用列表、详情、全屏编辑三层页面状态。
- `record` 与 `mode=detail|compare|edit` 写入 URL，支持浏览器和 Android 系统返回键退栈。
- 对照最多两条，只保存在当前 UI 会话；刷新后不会冒充持久化推荐结果。
- 表单先展示公司、岗位和状态；岗位事实、标签和反馈摘要按需展开。

### 不采用

- 招聘 Kanban：当前状态模型不足以支撑真实 pipeline，拖拽只会制造玩具感。
- 统计优先仪表盘：记录数量和状态分布不能替代逐条事实判断。
- AI 推荐排序：当前没有足够结果数据，也会违反人工确认边界。

## 交互协议

| 场景 | 行为 |
|---|---|
| 空列表 | 解释最小记录要求，并提供新增入口。 |
| 选择记录 | 桌面更新右侧详情；移动进入独立详情视图。 |
| 选择两条 | 进入事实对照；第三条被阻止并显示“最多比较 2 条”。 |
| 移动对照 | 逐字段显示 A/B 两条事实，不使用横向滚动表格。 |
| 编辑 | 移动端使用全屏编辑器；软键盘打开时隐藏底栏。 |
| 返回 | Android 首次返回关闭键盘，再次返回从编辑退到同一记录详情。 |
| 删除 | 必须确认；删除后同步移出对照集合。 |
| 导出 | 仅在浏览器确实生成下载时报告成功，否则显示失败。 |
| 保存 | 新记录写入当前任务 Evidence Gate，并回到刚保存的详情。 |

## 当前边界

- 当前清单只覆盖 sprint 最近记录，不能称为跨周完整机会资料库。
- 对照选择不跨刷新保存。
- 未实现未保存修改确认；后续在全局表单协议中统一处理。
- 生产构建仍有单 chunk 超过 500 kB 的告警，留到 P4 用真实启动指标处理。
