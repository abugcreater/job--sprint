# AI 运行恢复提示方案比较

日期：2026-07-24

| 方案 | 做法 | 优点 | 风险与结论 |
|---|---|---|---|
| A. 只优化文案 | 保留当前通用“AI 运行失败”文案。 | 改动最小。 | 无法区分登录、provider 与 API 层问题，不采用。 |
| B. 前端结构化分类 | runtime client 保留可公开的失败码，Coach 与 Stats 复用诊断映射。 | 用户能获得正确恢复动作，改动窄且可自动测试。 | 不处理真实 provider 治理，推荐。 |
| C. 自动重试和熔断 | 前端或服务端自动重试、退避、熔断并展示状态。 | 可减少短暂失败的人工操作。 | 需要 provider 限流、成本、幂等和线上证据，超出本轮范围。 |

裁决：采用方案 B。它不把一次本地 UI 改动伪装成 provider 可靠性治理，但先消除最容易误判的用户体验问题。
