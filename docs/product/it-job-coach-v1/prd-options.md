# PRD 方向版本评审

日期：2026-07-06

## 背景

Job Sprint 当前已有今日任务、学习、面试、机会跟进、Evidence Gate、复盘、AI 评分、React Web、Android WebView、Rust/SQLite 和服务端 HTTP 演示能力。上一轮 P0 暴露的问题是：页面能点，但流程没有输入、反馈、保存和读回。新一轮产品方向必须把“完整闭环”和“可验证工作流”作为硬约束。

目标定位调整为：面向泛 IT 求职者的 AI 求职教练与执行工作台。

## 版本 A：保守个人闭环版

### 定位

继续服务单用户，把个人求职闭环补完整。

### MVP 范围

- 用户目标画像：岗位方向、地区、薪资预期、时间投入。
- 知识边界：用户手动维护掌握程度、薄弱点、证据链接。
- AI 建议草稿：只生成今日建议、候选题目、知识卡片，不改正式日程。
- 结果复盘：记录投递、沟通、面试、失败原因、下一步动作。
- 有效推进数：每周统计真正推动求职的动作。

### 优点

- 风险最低，最符合当前代码和验收能力。
- 能先证明 AI 建议是否真的推动用户进展。
- 不需要立刻重做多用户数据模型。

### 缺点

- 产品吸引力偏弱，仍像个人工具。
- 很难验证泛 IT 人群差异。
- 用户增长和复用能力有限。

### 适用条件

当目标是尽快恢复稳定、避免大范围回归时选它。

## 版本 B：泛 IT 多用户版

### 定位

面向泛 IT 求职者的邀请制 AI 求职执行工作台。每个用户维护自己的日程、知识边界和目标岗位，系统给出个性化建议。

### MVP 范围

- 邀请制多用户登录。
- 每用户独立画像、日程、知识边界、机会记录和复盘。
- AI 生成三类 artifact：知识卡片、日程建议、候选题目。
- 生成物必须可接受、编辑、拒绝，并保留反馈。
- Evidence Gate 从“证据列表”升级为“行动门禁”：证据不足时建议先补证据再投递或面试。
- 每周有效求职推进数作为北极星指标。

### 优点

- 符合用户要求的方向调整。
- 能最大利用现有学习、面试、机会、复盘、AI 评分模块。
- 多用户扩展可控，不直接做公开 SaaS。

### 缺点

- 需要改数据模型和缓存策略，存在用户数据串线风险。
- 需要补 LLM artifact 持久化和评估门禁。
- UI 信息架构要重新围绕“用户画像 -> AI 建议 -> 执行 -> 复盘”组织。

### 适用条件

推荐采用。它在产品吸引力、当前能力复用和实现风险之间最平衡。

## 版本 C：AI 原生教练版

### 定位

以大模型为核心的求职 agent：自动读取用户画像、JD、复盘、知识边界，主动规划、追问、调度和复盘。

### MVP 范围

- 多轮 AI 导师对话。
- 自动拆解岗位要求。
- 自动生成周计划和日计划。
- 自动调整日程和知识边界。
- 自动追问用户复盘和面试表现。
- 跨用户匿名 benchmark。

### 优点

- 产品差异化最强。
- 更像真正的 AI coach，而不是套 AI 按钮的日程工具。
- 对未来商业化和模板路径包更友好。

### 缺点

- 当前证据不足，容易变成伪 AI。
- 自动改日程和自动判断能力缺口风险高。
- 隐私、幻觉、误导、成本和延迟都更难控制。
- 之前 P0 问题说明团队还不能跳过基础闭环直接做高自治 agent。

### 适用条件

作为 P2/P3 愿景保留，不作为本轮主 PRD。

## 团队裁决

推荐版本：版本 C 的产品叙事 + 版本 B 的工程边界。

也就是：对用户表达为“AI 原生求职教练”，但 MVP 只做小范围邀请制、多画像/多用户、严格证据门禁和人工确认，不做自动 agent 或公开 SaaS。

裁决理由：

1. 用户真正要的吸引力来自 AI 教练，而不是再做一个多用户日程系统。
2. 当前工程不能直接承受高自治 agent，所以 AI 只能生成草稿建议，用户确认后进入正式计划。
3. 泛 IT 用户差异较大，MVP 先用角色族和目标画像收敛，而不是一次做全行业题库。
4. 现有学习、面试、机会、复盘、Evidence Gate、AI 评分能力都能被复用到 AI 教练闭环。
5. “有效求职推进数”和 AI 建议采纳完成率能证明产品是否真的推动求职。

## 明确不选

- 不选版本 A 作为最终方向，因为它无法回答“如何让不同用户添加自己的日程和知识边界”。
- 不选完整版本 C 作为本轮，因为自动改日程、长期后台 agent、跨用户知识池会过早放大隐私、幻觉和自动化风险。

## 最终 PRD 名称

Job Sprint 泛 IT AI 求职教练 PRD v1.0。
