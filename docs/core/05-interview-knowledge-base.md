## RAG 防幻觉与 no-answer

- 分类：RAG
- 来源类型：project

- 面试问题：RAG 为什么不能完全消灭幻觉？
- 60 秒回答：RAG 的价值是让模型回答前先拿到外部知识证据，并把答案和来源绑定起来，所以能降低凭空编造。但检索可能漏召回、召回内容可能过期或不相关，模型仍可能误用证据。因此生产系统要有 citation、score threshold、no-answer、权限过滤和评测。
- 3 分钟回答：可以从检索、重排、证据拼接、生成、引用、拒答和评测七步讲。当前项目是最小演示，生产化升级会补 embedding/vector store、hybrid search、rerank、ACL filter、eval 和审计。
- Java 映射：检索服务、ACL filter、ES/BM25/向量检索、结果可信度
- 项目证据：data/knowledge/供应商规则.md 与供小慧 Agent 知识回答链路
- 风险：把关键词检索说成完整向量 RAG。
- 不能说：已上线大规模向量库；RAG 完全解决幻觉
- 安全表达：当前是最小检索增强样例，生产化可升级为混合检索和评测体系

## MQ 不丢不重与幂等消费

- 分类：Spring / MySQL / Redis / MQ 经验
- 来源类型：derived
- 可公开总结：可靠消息要分生产、Broker、消费、重试、DLQ、幂等和观测，不承诺绝对不重。
- 面试问题：如何保证 MQ 消息不丢不重？
- 60 秒回答：工程上通常不能承诺绝对不重，所以核心是至少一次投递 + 消费幂等。生产端要确认发送结果，Broker 要持久化，消费端要手动 ACK、失败重试和 DLQ。业务侧用唯一键、版本号、状态机或幂等表防重复影响，最后用消费延迟、失败率和积压指标监控。
- 3 分钟回答：结合缓存构建类场景讲：消息解析、业务 key、版本号防乱序、分布式锁或 CAS 防并发写、重试和死信处理、指标告警。
- Java 映射：RabbitMQ/RocketMQ/Kafka、ACK、重试、DLQ、幂等
- 项目证据：可映射 searchblackhole/searchpackage 的 MQ 消费和缓存构建口径。
- 风险：说 exactly-once 而无法解释业务幂等。
- 不能说：MQ 天然不重复；只靠 Redis 覆盖就是完整幂等
- 安全表达：至少一次 + 业务幂等 + 监控补偿

## Spring 事务、MySQL、Redis 高频追问

- 分类：Spring / MySQL / Redis / MQ 经验
- 来源类型：derived
- 可公开总结：必须能讲事务传播、自调用失效、MVCC、索引、锁、缓存一致性、热点 key、分布式锁。
- 面试问题：Spring 事务为什么会失效？缓存和数据库怎么保证一致？
- 60 秒回答：事务失效常见原因是自调用绕过代理、方法非 public、异常被吞、传播行为选错、异步线程脱离事务上下文。缓存一致性通常不追求强一致，而是更新 DB 后删除缓存，配合重试、延迟双删或消息补偿，并通过 TTL 和监控兜底。
- 3 分钟回答：补充 REQUIRED/REQUIRES_NEW/NESTED 的边界、连接池风险、MVCC 快照读/当前读、Redis 分布式锁过期和续期风险。
- Java 映射：Spring AOP、事务传播、MySQL MVCC、Redis 缓存治理
- 项目证据：可绑定搜索缓存服务、AI 服务化鉴权和任务状态。
- 风险：只背传播级别，不会讲失效场景。
- 不能说：Redis 锁绝对可靠
- 安全表达：通过事务边界、幂等和补偿降低不一致影响

- 分类：API 服务化
- 来源类型：project

- 面试问题：为什么不直接让前端调大模型 API？
- 60 秒回答：因为 token、权限、成本和业务边界都不能放在前端。后端服务化后，可以统一做 API Key/JWT、用户上下文、参数校验、超时重试、工具权限、日志审计、metrics 和降级兜底。这样 AI 只是后端链路里的一个外部能力，而不是裸露的第三方调用。
- 3 分钟回答：可以按入口、鉴权、业务编排、模型/工具调用、可观测、异常兜底、成本控制讲，并映射到 Spring Boot Controller、Service、Client、Filter、Actuator、Micrometer。
- Java 映射：Spring Boot 服务化、鉴权、观测、外部依赖治理
- 项目证据：src/replay/api.py、auth.py、observability.py
- 风险：只讲 prompt，不讲服务治理。
