# Job Sprint 项目全景介绍

日期：2026-07-22
文档定位：面向产品协作者、技术评审者和面试交流的项目总览。它汇总当前事实源，不替代 PRD、架构设计、验收报告或发布记录。

## 1. 一句话说明

Job Sprint 是一个面向泛 IT 求职者的 AI 求职教练与执行工作台。它把“我想找什么工作、还缺什么能力、今天该推进什么、机会和面试进展如何、AI 建议是否真的有效”串成可记录、可确认、可复盘的个人求职闭环。

它不是让 AI 替用户投递简历的自动化工具，也不是招聘方使用的 ATS。当前采用小范围邀请制，目标是在 2 到 8 周的集中求职、转岗或跳槽场景中，帮助用户把模糊的准备过程变成可持续推进的行动。

## 2. 项目背景与问题

### 2.1 起点

项目最初是个人高级 Java 后端求职冲刺工作台。继续沿用“只服务一个既定技术画像”的设计，会出现两个问题：

1. 新用户会被旧用户的学习内容、面试题和默认数据误导，无法形成自己的准备路径。
2. 求职活动分散在简历、笔记、日历、招聘平台和聊天记录中，用户知道自己很忙，却很难判断今天的行动是否真正推动了求职。

因此项目被重构为面向泛 IT 从业者的产品。首批覆盖后端、前端、测试、运维、数据、移动端、产品、项目、实施和技术支持等角色族，而不是把某一个 Java 工程师的历史数据包装成所有人的默认路径。

### 2.2 要解决的核心问题

| 用户问题 | Job Sprint 的处理方式 |
|---|---|
| 目标岗位、简历证据和当前能力之间没有统一视图 | 通过求职画像、简历素材和知识边界建立个人上下文。 |
| 每天不知道优先做什么 | 以目标岗位、机会状态、可用时间和能力缺口生成今日行动，并允许用户调整。 |
| 学习、面试和投递彼此断开 | 将知识卡、候选题、机会、复盘和任务放在同一条个人进度链中。 |
| “完成”常常只是勾选，没有产出 | 用 Evidence Gate 要求笔记、答案、评分、反馈或机会状态等可核验结果。 |
| AI 建议难以信任，也无法改善 | AI 只生成草稿；用户必须采纳、编辑或拒绝，系统记录来源、原因、运行状态和反馈。 |
| 邀请制场景下容易看到或写入他人数据 | 使用认证、`dataScope` 和服务端 scope 约束；普通用户与 owner 管理能力分开。 |

### 2.3 当前阶段

当前阶段是“准生产演示级 / 小范围邀请制产品化迭代中”。它已经具备可用的多端工作流、数据隔离与验证基础，但产品定位和交付边界保持收敛：先证明一个普通用户能完成自己的求职闭环，再讨论公开注册、组织协作或商业化扩张。

## 3. 面向用户的产品闭环

Job Sprint 的核心不是模块堆叠，而是一条可重复执行的闭环：

```text
简历 / JD / 目标岗位 / 可用时间
              |
              v
      求职画像与知识边界确认
              |
              v
      今日行动与优先级建议
              |
              v
学习、面试训练、机会跟进、复盘
              |
              v
      Evidence Gate 提交证据
              |
              v
AI 草稿 + 用户采纳/编辑/拒绝 + 运行记录
              |
              v
  周期复盘、结果归因、下一轮行动调整
```

### 3.1 普通用户的主要能力

| 能力域 | 用户可以做什么 | 在闭环中的作用 |
|---|---|---|
| 首次建档与画像 | 导入或填写简历素材，确认目标角色、工作年限、偏好、可用时间和求职状态。 | 建立 AI 与规则建议的个人上下文。 |
| 知识边界 | 维护已掌握、待补齐和需要证明的技能主题，查看来源和采纳反馈。 | 防止建议脱离真实能力，识别需要补的缺口。 |
| 今日行动 | 查看和调整当天任务、延期记录和优先级。 | 将长期求职目标拆成可执行的当天动作。 |
| 学习与知识卡 | 记录学习主题、笔记、重点和薄弱点。 | 为能力增长留下可核验的证据。 |
| 面试训练 | 维护候选题、回答、评分、错题和追问。 | 把“准备面试”转为可复盘的输出。 |
| 机会跟进 | 记录职位、JD 信号、投递、沟通、面试和结果状态。 | 将真实机会进展带回任务优先级。 |
| 复盘与统计 | 查看完成情况、AI 反馈、机会结果和阶段性归因。 | 判断行动是否真的推进了求职，而非只制造忙碌感。 |
| 账户、同步与备份 | 查看当前账号和数据域，进行个人同步、导出和恢复。 | 降低换设备或本地数据丢失的风险。 |

### 3.2 Evidence Gate

Evidence Gate 是项目的产品约束，而不是装饰性的“完成”按钮。不同活动应有对应证据：

| 活动 | 最小证据示例 |
|---|---|
| 学习 | 笔记、知识卡或可回看的要点。 |
| 面试训练 | 回答、评分、错题或复盘结论。 |
| 机会跟进 | 投递、沟通、面试或结果状态。 |
| AI 建议 | 用户的采纳、编辑或拒绝，以及原因。 |

证据不足时，产品应提示补充，而不是把任务误记为已完成。这样复盘看到的是有效进展，而不是简单的勾选数量。

### 3.3 AI 在产品中的位置

AI 是教练助手，不是自动执行者。当前可生成知识卡、日程建议、候选题和下一步行动等 artifact；建议会带有输入摘要、来源、理由、schema 状态和运行记录，并支持用户反馈。

AI 的工作边界如下：

- 只能生成待确认草稿，不能直接修改正式日程、机会或用户档案。
- 没有足够的画像、知识边界或机会信息时，应追问或输出 `unknown`，不能假装了解用户。
- 必须区分事实、推断和建议，不能编造项目经历、公司、面试结果、指标或用户技能。
- provider 未配置、不可用或返回不合格结果时，系统会明确记录降级原因，并使用本地规则化 fallback；这不应被展示为真实大模型成功。
- 用户的采纳、修订、拒绝和原因会进入反馈与运行审计，供后续复盘和产品改进使用。

## 4. 权限、数据与角色边界

### 4.1 为什么会有“邀请用户”

邀请不是求职画像的一部分。它属于小范围试用阶段的账号开通与观察机制：owner 可以管理邀请批次、首登完成情况和账号生命周期；普通用户只处理自己的画像、求职行动和个人备份。

产品上已将 owner 能力从普通用户的“更多”导航中收敛到管理员中心，避免新用户把账号管理误解为自己的求职任务。邀请机制的价值是控制试用范围、验证首登闭环和维护账号状态，不是给每个用户增加后台功能。

### 4.2 数据隔离模型

隔离不依赖前端“看不见就算隔离”。当前模型包含：

1. 登录会话或 bearer token 识别当前账号。
2. 服务端为账号解析 `dataScope`，读取和写入 runtime、画像、机会、复盘、AI artifact 与反馈时按 scope 约束。
3. React 本地缓存带有数据所属账号标记；账号切换或本地 `storageOwner` 不匹配时，不允许把旧用户缓存优先上传到新账号。
4. Rust/SQLite 将 runtime items、`llm_runs` 和 `llm_feedback` 按用户 scope 持久化；Node 兼容路径也保持同样的 scope 合同。
5. 普通用户导航不暴露 owner 管理入口；服务端权限仍是最终裁决，不能只依赖前端隐藏。

这套设计解决的是邀请制小规模多账号的数据边界。它还不是组织租户、成员协作、计费或细粒度企业权限系统。

### 4.3 安全边界

- provider token、服务器账号、session secret、SQLite 数据库、签名材料和真实交付配置均不进入公开仓库。
- Web 会话 cookie 使用 `HttpOnly` 与 `SameSite=Lax`；HTTPS 或代理标记下使用 `Secure`。
- Android 远端地址要求 HTTPS，关闭 cleartext，并受 host allowlist 约束；远端不可用时才回落到本地安全资源。
- Android 凭据使用 Android Keystore 与 AES/GCM 保存，旧明文键会迁移和清理。
- public-safe 构建和敏感扫描用于防止本机路径、内部线索、密钥、私钥和部署信息进入公开包。

## 5. 技术架构

### 5.1 总体结构

```text
                         +-----------------------+
                         |  Anthropic-compatible |
                         |  provider (optional)  |
                         +-----------+-----------+
                                     |
                                     v
+----------------+        +-----------------------+        +------------------+
| React Web      | <----> | Rust + Axum + SQLite  | <----> | SQLite runtime   |
| TypeScript     |        | primary runtime path  |        | scoped records   |
+-------+--------+        +-----------------------+        +------------------+
        |                         ^
        |                         |
        |                 +-------+----------------+
        +---------------> | Node.js compatibility  |
                          | auth/static/runtime API |
                          +------------------------+

+----------------+        +------------------------+
| Android Java   | -----> | remote HTTPS React     |
| WebView shell  |        | or local fallback      |
+----------------+        +------------------------+
```

运行时的主数据路径是 React Web 与 Rust/SQLite。Node.js 仍保留认证、静态资源和 `/api/runtime` 等兼容 API，用于兼容、回滚和轻量演示。Android 不重复实现业务页面，而是承载远端优先的 React WebView，并在远端不可用时回退到本地 React assets 和旧 public-safe 资源。

### 5.2 技术选型与理由

| 层 | 技术选型 | 为什么这样选 | 当前取舍 |
|---|---|---|---|
| Web | React 19、TypeScript、Vite | React 适合把画像、任务、机会和复盘拆成可复用交互页面；TypeScript 约束数据形状；Vite 让本地开发、测试和 Android 资产构建保持轻量。 | 不是服务端渲染站点，首要目标是受登录保护的工作台交互。 |
| 状态与路由 | Zustand、React Router、Hash Router | Zustand 适合小到中等规模的前端状态和本地持久化；Hash Router 同时适配普通 Web 与 Android WebView 静态入口。 | 本地状态必须与账号所有者和服务端同步规则一起使用，不能把 localStorage 当成权限边界。 |
| UI | Tailwind | 便于在同一组件体系内统一移动端、桌面端、空态、表单反馈和权限可见性。 | 设计系统仍在迭代，体验改动需配合实际页面验收。 |
| Node 兼容层 | Node.js CommonJS + 原生 HTTP 路由 | 延续现有服务入口，提供登录、静态资源、JSON runtime 兼容 API 和 AI fallback，改造成本低。 | 它不是长期的高并发核心数据层，新的持久化能力优先走 Rust。 |
| 后端主路径 | Rust、Axum、SQLx、SQLite | Rust 适合以单二进制、低资源方式交付服务；Axum 提供明确的 HTTP 路由与中间件边界；SQLx 和 SQLite 让邀请制小规模用户的持久化、迁移和 scope 校验可测试。 | SQLite 合适于当前小范围邀请制，不等于组织级高并发多租户数据库方案。 |
| 数据存储 | localStorage、`/api/runtime`、SQLite | localStorage 改善离线和首次打开体验；API 负责同步；SQLite 负责服务端持久化、用户 scope 和 AI 审计。 | 导入/恢复目前仍有部分独立 localStorage 数据未进入统一备份格式，需要继续收敛。 |
| AI 集成 | Anthropic-compatible provider + 本地 fallback | 可适配不同兼容 provider，同时在 provider 不可用时保持基本教练流程可用；统一 schema、输入 hash、运行记录和反馈，避免“黑盒文案”。 | fallback 是降级能力，不能替代真实模型质量、成本和稳定性治理。 |
| Android | Java + WebView | 复用已验证的 React 业务体验，降低为每个模块重新开发原生页面的成本，同时保留远端 URL、会话和本地 fallback 的原生控制点。 | WebView 不是完整原生重写；远端登录、cookie 和真机网络链路需持续验收。 |
| 质量门禁 | Vitest、Node tests、Playwright、Cargo tests、Android/交付脚本 | 覆盖前端交互、API 合同、Rust/SQLite、浏览器、Android 读回、敏感扫描和交付证据，让“功能能点”不成为唯一标准。 | 本地测试、Git 合并、真实服务器和 Android 正式交付是不同验收层，不能互相替代。 |

## 6. 如何开发和交付

### 6.1 需求到合并的开发方法

项目采用“产品闭环优先 + 小步可验证”的开发方式：

1. **确认事实源**：先查看当前 PRD、产品账本、已知问题、验收风险和 Git 状态，避免把历史描述当成当前结论。
2. **定义用户闭环**：每个需求先说明目标用户、触发条件、用户操作、数据变化、权限边界、空态和验收结果。产品级改动建立 feature capsule，不直接在代码里堆功能。
3. **从 `develop` 创建单意图分支**：功能、修复、测试、文档和仓库治理分别使用对应短分支，避免无关改动混在一个提交中。
4. **前后端同时约束**：涉及账号、数据或 AI 时，前端可见性、API 鉴权、`dataScope`、SQLite 记录和回归测试一起修改；不把前端过滤当作数据隔离。
5. **按影响范围验证**：Web 使用类型检查、Vitest 和浏览器测试；Node/Rust 使用 API 与契约测试；Android 使用构建、功能流和重启读回；变更发布链路时额外跑交付门禁。
6. **安全检查与 PR 收口**：提交前执行 diff 检查、GitFlow 策略和敏感扫描；PR 必须通过 required check，squash merge 到 `develop` 后删除远端与本地短分支。

### 6.2 GitFlow 与发布边界

代码仓库使用 `main + develop + 短生命周期分支`：

| 分支 | 含义 |
|---|---|
| `main` | 已验收、可发布、可打标签的稳定源码版本。 |
| `develop` | 下一版本的日常集成基线。 |
| `feature/*`、`fix/*`、`test/*`、`docs/*` 等 | 从 `develop` 创建，完成后只能通过 PR 合入 `develop`。 |
| `release/vX.Y.Z` | 仅用于从 `develop` 到 `main` 的发布冻结、发布说明和阻断修复。 |

“已提交”“已推送”“已创建 PR”都不算完成；普通需求完成的标准是 PR 已合入 `develop`、短分支已删除、工作树干净。发布后还需把 `main` 回同步到 `develop`。

### 6.3 代码发布不等于服务器已更新

这是项目最容易被误解的边界，必须单独说明：

```text
开发分支合入 develop
        !=
release 合入 main 并打 tag
        !=
构建服务器交付包
        !=
同步、重启、远端登录/保存读回证据
        !=
Android 正式 APK 与远端真机验收
```

服务器交付需要额外构建 Linux x86_64 Rust 二进制、public-safe 静态资源与交付 manifest，执行同步并证明远端 manifest hash、登录、session、保存和读回。Android 正式交付还需要仓库外签名材料、APK hash、签名验证以及远端 HTTPS 真机证据。

这些动作需要明确授权和私有交付环境，不能因为 GitHub PR 合并就自动发生。这样既避免将本地验证冒充生产交付，也避免把密码、token、签名材料或真实用户数据写进仓库。

## 7. 验收状态与当前限制

### 7.1 已有验证基础

当前核心文档记录了以下已验证范围：

- React 主工作流、普通用户导航、画像保存反馈、统计页和新用户空态有类型检查、组件测试和浏览器验证。
- Node 与 Rust 均覆盖认证、`dataScope` 隔离、runtime 合同、AI artifact/feedback 和 SQLite 写入等关键路径。
- Android 本地 WebView 覆盖保存、延期、导入、重启读回和移动端布局；远端路径另有独立验收要求。
- AI 运行有 schema、provider/fallback、输入摘要、反馈和审计记录，能够把 `provider_not_configured`、timeout 或 fallback 与真实 provider 成功区分开。
- 仓库有 GitFlow、产品迭代、敏感扫描、public-safe、功能覆盖和交付 readiness 等门禁，避免把局部测试通过扩大解释为完整交付。

### 7.2 编写本文档时的版本事实

截至 2026-07-22：

- `main` 的稳定标签为 `v0.2.2`。
- `develop` 已包含两项后续本地运行时验证补强：经 Vite 代理的 Node 登录诊断，以及经 Vite 代理的 Rust 认证与 `llm_runs` 读回诊断。
- 上述 `develop` 变更已通过 GitHub PR 合入源码集成线，但尚未因此自动构建、同步或重启远端服务器。任何服务器版本判断必须以新的交付 manifest、远端 health、认证和数据读回 evidence 为准。

换句话说，当前可以证明代码与本地自动化验证已经前进，但不能把这两项新增变更描述成“服务端已经同步”。

### 7.3 仍需正视的限制

| 范围 | 当前限制 | 后续方向 |
|---|---|---|
| AI provider 可靠性 | 已有 timeout 与 fallback，但错误分类、重试、熔断、成本监控和面向用户的提示仍不完整。 | 补 provider 观测、策略和体验反馈。 |
| AI 质量归因 | 已记录 artifact 与用户反馈，跨周的服务端长期归因和真实岗位深度解析仍有限。 | 将机会状态、面试结果和反馈聚合到可解释的周复盘。 |
| 多用户范围 | 已覆盖邀请制账号和 scope 隔离，不包含公开注册、组织租户、协作或计费。 | 只有验证个人闭环后才评估扩展。 |
| Android 远端体验 | 本地 WebView 有充分验证，但新 APK、域名、证书或远端服务变化都会使旧远端 evidence 失效。 | 变更后重新跑 HTTPS 首访、登录、保存和重启读回。 |
| 备份完整性 | “更多”页的导入恢复已覆盖 React 主状态，部分独立 localStorage 数据仍需统一。 | 设计版本化的聚合备份格式。 |
| 交付状态 | 历史服务器/Android evidence 只能证明对应构建当时的状态。 | 每次源码、Rust 二进制、静态资源、APK 或证书变化后重新生成关联证据。 |

## 8. 项目边界与非目标

为避免对外介绍夸大范围，以下边界必须保留：

- 不是公开 SaaS，也没有开放注册、付费、组织租户或协作后台。
- 不是企业 ATS、招聘方管理系统或招聘流程外包平台。
- 不执行自动投递、批量海投、代替用户做求职决策或伪造求职材料。
- 不把 AI 当成自动写库、自动排程的执行主体；用户确认和证据门禁始终存在。
- 不是通用学习平台或无限题库；内容服务于用户的目标岗位和真实机会。
- 不是以 Spring Boot、MyBatis、Redis、MQ 为主线的高级 Java 后端项目；若用于 Java 后端求职，需要另有对应技术主线证据。
- 不把全局 AI 团队能力伪装成仓库内部强隔离 multi-agent runtime；项目只保留调用边界、审计和验证方式。

## 9. 对外介绍模板

### 9.1 30 秒版本

“Job Sprint 是一个面向泛 IT 求职者的 AI 求职教练工作台。它把用户画像、能力边界、每天的求职行动、机会进展、面试训练和复盘证据放到同一条闭环里。AI 只给可追溯的草稿，用户必须确认或反馈；后端通过账号 scope 做数据隔离。技术上用 React 工作台、Rust/SQLite 主运行时、Node 兼容层和 Android WebView，重点是让一个普通用户能持续、可信地推进求职，而不是做自动投递或公开 ATS。”

### 9.2 适合技术交流的展开方式

1. 先讲产品问题：求职准备分散，缺乏每天的优先级和可复盘证据。
2. 再讲闭环：画像和知识边界决定建议，今日行动驱动学习、面试与机会跟进，Evidence Gate 和 AI 反馈回到下一轮复盘。
3. 说明可信边界：邀请制、服务端 `dataScope`、AI 草稿确认、provider/fallback 区分、运行审计。
4. 最后讲工程取舍：React 复用到 WebView；Rust/SQLite 负责可验证的持久化与权限；Node 保留兼容入口；GitFlow、测试和交付 evidence 防止把本地成功误说成线上发布。
